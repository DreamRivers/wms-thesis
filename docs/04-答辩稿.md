## 二、PPT 讲解(8 分钟,约每页 40 秒)

### P3 选题背景(40 秒)

随着电子商务的快速发展,我国 2025 年电商交易额已突破 15 万亿元。然而,我国仓储费用占 GDP 比重约为 5%,远高于发达国家 2%~3% 的水平,说明仓储管理效率仍有较大提升空间。传统人工管理存在效率低、错误率高、库存不透明等问题,已无法满足现代电商"快速响应、零差错、可追溯"的要求。

### P4 研究意义(40 秒)

一套高效的 WMS 系统,可将仓储人力成本降低 30%~50%,提升盘点准确率至 99% 以上,缩短出入库作业时间 40%,并通过预警机制将呆滞料损失减少 60%。因此,本课题的研究具有重要的实践价值。

### P5 国内外现状(40 秒)

国外 WMS 以 Manhattan、HighJump 为代表,产品成熟但价格昂贵。国内有富勒、科箭、旺店通等本土厂商,产品多 SaaS 化,但对中小企业而言,定制化与价格仍是痛点。本课题旨在基于开源框架自研一套轻量级 WMS,填补这一空白。

### P6-P7 需求分析(60 秒)

系统支持 4 类角色:系统管理员、仓库管理员、部门负责人、普通员工。
核心功能包括 7 大模块:用户管理、商品管理、库位管理、入库管理、出库管理、库存管理、预警与报表。

### P8 业务流程(60 秒)

入库流程:仓管创建草稿 → 提交 → 主管审核 → 执行 → 完成,共 7 个状态。
出库流程:员工申请 → 部门负责人审核 → 仓管审核 → 拣货出库 → 完成,采用两级审批流。

### P9 技术栈(40 秒)

后端采用 Spring Boot 2.7.18 + MyBatis-Plus + Sa-Token + MySQL 8.0 + Redis;
前端采用 Vue 3 + Vite 5 + Element Plus + ECharts + Pinia;
采用前后端分离架构,后端 RESTful API,前端 SPA。

### P10-P11 系统设计(60 秒)

系统采用经典三层架构。数据库设计 13 张核心表,关键设计包括:实时库存 `wms_stock` 联合唯一索引 `(goods_id, location_id, batch_no)`,所有单据采用"主表+明细表"双表结构,公共字段 `create_time/update_time/deleted` 统一管理。

### P12-P17 系统实现(180 秒,每页 30 秒)

登录采用 Sa-Token 实现 Token 鉴权,密码 BCrypt 加密。

入库单状态机:`DRAFT → PENDING → APPROVED → EXECUTING → FINISHED`,支持驳回与作废。
关键代码使用 `SELECT ... FOR UPDATE` 行级锁,在同一事务内完成"写库存+写流水"。

出库采用两级审批,部门负责人先审,仓管再审,每次审批写入 `wms_outbound_approval` 留痕表。

库存预警使用 `@Scheduled` 定时任务,每小时扫描低库存与临期商品,写入 `wms_notification` 表。

报表使用 ECharts 实现 4 张可视化图表:出入库趋势、库存结构、TOP10 商品等。

### P18 系统测试(40 秒)

完成 12 项功能测试用例,均通过;性能测试使用 JMeter 模拟 100 并发,关键接口响应 ≤ 240ms;兼容 Chrome、Edge、Firefox 三大浏览器。

### P19 创新点(40 秒)

本系统的创新点包括:1) 状态机规范化单据流转;2) 行级锁+事务保证库存并发安全;3) 定时任务实现两级预警;4) AOP 自动记录操作日志;5) RBAC 精细化权限控制。

### P20 总结与展望(30 秒)

本文完成了一套可运行的 WMS 系统,涵盖 7 大模块、13 张数据表、25+ 业务页面。未来可在微服务化、AI 补货建议、移动端扫码、IoT 集成、多租户 SaaS 等方向进一步优化。

## 四、评审常见 15 问及应答

### Q1:为什么选 Sa-Token 而不是 Spring Security?
A:Spring Security 配置复杂,学习成本高,适合复杂场景;Sa-Token 配置简单,内置 RBAC + JWT + Redis,API 友好,适合毕设规模。

### Q2:库存并发安全如何保证?
A:三个层面:1) 数据库 `SELECT ... FOR UPDATE` 行级锁;2) `@Transactional` 事务保证原子性;3) 写库存+写流水在同一事务。

### Q3:为什么出库要两级审批?
A:业务上,普通员工领用物资应先经本部门负责人审批,确认业务合理性,再由仓管确认库存可用性,避免越权与库存滥用。

### Q4:如何设计状态机?
A:在 Service 层封装 `transition` 方法,根据当前状态判断允许的下一状态,非法状态抛 `BizException`。所有 controller 入口都通过 Service 校验。

### Q5:为什么不选 MyBatis 而选 MyBatis-Plus?
A:MyBatis-Plus 是 MyBatis 的增强,提供 BaseMapper 通用 CRUD、LambdaQueryWrapper、分页插件等,减少 80% 重复代码,且 100% 兼容 MyBatis。

### Q6:流水表和库存表的关系?
A:库存表存当前量,流水表存每次变动(before/after/操作人/单号)。流水表只追加,用于审计与回溯。

### Q7:定时任务如何防止重复扫描?
A:每个预警任务先去 `wms_notification` 查当天是否已生成同类型同 ref_id 通知,避免重复入库。

### Q8:前后端分离的优缺点?
A:优点:职责清晰、易扩展、可多端复用;缺点:SEO 不友好、首屏加载慢(可 SSR 解决)。本系统为内部系统,优先开发效率。

### Q9:本系统可以承载多少并发?
A:理论 MySQL 8.0 + Redis 7 + Tomcat 线程池 200 可支撑 500~1000 并发(读多写少场景),实际需根据硬件压测。

### Q10:为什么用 Vite 而不是 Webpack?
A:Vite 基于 ESM 原生模块,启动 < 1 秒,HMR 几乎瞬时;Webpack 需打包,大型项目启动慢。Vite 5 已稳定,适合现代前端开发。

### Q11:Sa-Token 的 Token 存哪里?
A:默认存内存,本系统配置 sa-token-redis-jackson 存 Redis,实现分布式环境下的 Token 共享与失效控制。

### Q12:如何防止 XSS 与 SQL 注入?
A:XSS:Vue 默认转义;SQL 注入:MyBatis-Plus 使用 #{} 占位符预编译;系统不使用拼接 SQL。

### Q13:为什么 E-R 图不画所有字段?
A:答辩时画完整 13 张表会非常拥挤,只画核心实体关系(7~8 个),其他用文字说明。详细字段在论文附录。

### Q14:工作量和难度如何?
A:本系统 2 人 8 周工作量,代码量约 8000 行,涵盖前后端全栈。难度主要在业务理解与状态机设计。

### Q15:系统的商业价值?
A:可直接为中小型电商部署,替代人工管理;也可作为 SaaS 化产品的基础版本,通过多租户改造实现商用。
